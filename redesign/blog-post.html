Code Quality
Using Github
Actions

An additional layer of scrutiny so your code quality is practically bulletproof — enforced on every push, no escape hatches.

Published August 3, 2025
Reading Time 10 min
Author Allan Fernandes
Part of Code Quality Series

In my previous article I showcased how you could set up code quality checks locally and share that config across developers — but that whole setup had one flaw: it could be bypassed fairly easily.

The problem: Local hooks can be skipped with --no-verify. What you need is a CI-level check that has no escape hatch.

The straightforward solution is enforcing checks on the CI pipeline as well. This gives you an additional layer of scrutiny so your code quality is practically bulletproof.

Understanding Github Actions

To oversimplify, think of Github actions as automated processes you can set up in your repository. GitHub provides virtual machines (Linux, Windows, or macOS) that can run a series of commands, or Actions, for you.

You can use these actions to automatically build your code, run tests, and check formatting every time you push a change or open a pull request — streamlining your entire CI/CD pipeline.

Setting Up Your First Github Action

On your main branch, create a directory called .github and a sub-directory called workflows. This is where all your GitHub Actions will live.

Create a code-quality.yml file. The .yml extension is for YAML — the language used to define your workflow actions.

⚠️ YAML is painfully pedantic with formatting. One wrong indentation and you'll waste 30 minutes of your life. Save yourself the tears — ask an AI to cross-verify your config. 9 out of 10 times if it's not working, you've put an extra tab somewhere.

Copy the below into your code-quality.yml:

.github/workflows/code-quality.yml YAML 
name: Code Quality

on:
  push:
    branches:
      - '**'
  pull_request:
    branches:
      - '**'

jobs:
  lint:
    name: Code Quality
    runs-on: ubuntu-latest
    steps:
      - name: ⬇️ Checkout repo
        uses: actions/checkout@v4

      - name: 🔧 Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 'lts/*'
          cache: 'npm'

      - name: 📦 Install dependencies
        run: npm ci

      - name: 🚨 Run ESLint
        run: npx eslint . --max-warnings=0

      - name: 🔍 Run Prettier check
        run: npx prettier --check .

What Each Part Does

  • name: Code Quality — Sets the workflow's display name in the Actions tab of your repo.
  • on: — Defines trigger events. Runs on every push and pull request to any branch (**).
  • jobs: — Contains all the jobs in the workflow. lint is the unique ID for this job.
  • runs-on: ubuntu-latest — Executes on a GitHub-hosted Ubuntu VM.
  • actions/checkout@v4 — Checks out your repo so the job can access the code.
  • actions/setup-node@v4 — Sets up a Node.js environment with LTS and npm caching.
  • npm ci — Installs dependencies. Faster and more reliable than npm install in CI.
  • eslint . --max-warnings=0 — Fails the job if any linting warnings are found. Zero tolerance.
  • prettier --check . — Fails the job if files aren't formatted correctly, without modifying them.

Test Your Github Action

If you now push your commit up to your repo, it should trigger the GitHub Action. You can verify this has worked by going to your repository on GitHub, clicking the Actions tab, and checking that the Code Quality workflow shows up and has run successfully.

★
Pro Tip
Use the official Github Actions VS Code plugin to monitor all your actions running from the editor itself — no need to open the browser. You're welcome.

Enable Branch Protection Rules

The final piece of the puzzle is ensuring our Action is enforced across all branches — so no one can merge code that fails the check.

Go to Settings → Branches → Branch Protection Rules → Add rule and configure:

  • Branch name pattern: *
  • ✅ Require status checks to pass before merging
  • Search for Code Quality — this should appear as a Github Action
  • Hit Create

Testing It All Works

Create a new branch, push some intentionally rubbish code, and open a PR. The GitHub Action should trigger, fail, and block the merge. Everyone's happy, no one's wasting time arguing over single vs double quotes vs tab spacing vs semicolons — and we can all go back to maximising shareholder value.
